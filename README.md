# 顶绳攀岩跳一跳原型

## 项目目标

这是一个原生 HTML5 Canvas 2D 网页原型，用来验证“按住蓄力、松开出手、抓住岩点或失败、继续向上攀爬”的核心循环。玩法类似《跳一跳》的距离判断，但场景改为竖直室内攀岩墙。

当前版本重点是核心玩法和参考图风格的可运行视觉原型，不包含正式美术、音效、物理引擎或复杂工程架构。

## 文件结构

- `index.html`：页面入口，只加载 Canvas、样式和脚本。
- `style.css`：竖屏比例、自适应缩放、浅色页面背景和触控限制。
- `game.js`：状态机、输入、蓄力判定、无限岩点生成、角色动作、摄像机、成绩存储和 Canvas 绘制。
- `assets/pink_climbing_holds_complete_assets/`：粉色线路岩点 PNG 资产，Canvas 会优先使用 `individual_png` 内的单体透明图片绘制主攀爬线路。
- `assets/light_purple_climbing_holds_complete_assets/`：浅紫色非线路岩点 PNG 资产，用于区分墙面上的辅助/非线路岩点。
- `README.md`：项目说明。

## 操作方式

手机端：按住 Canvas 任意位置蓄力，松开后出手。失败结算后点击屏幕重新开始。

电脑端：用鼠标左键模拟触控，按住蓄力，松开出手。没有键盘操作。

输入统一使用 Pointer Events，包括 `pointerdown`、`pointerup`、`pointercancel`，并处理失焦和指针捕获丢失。

成功抓到岩点后，下一次按压可以立即开始蓄力；身体跟随、脚移动、休息甩手和少量残余镜头跟随会被快速收尾，不需要等完整表现动画播放完。

## 核心蓄力判定

蓄力值会在 0 到 1 之间往返循环，按住越久不再停在最大值，而是到头后向回递减：

```js
charge += chargeDirection * deltaTime / chargeDuration;

if (charge >= 1) {
  charge = 2 - charge;
  chargeDirection = -1;
} else if (charge <= 0) {
  charge = -charge;
  chargeDirection = 1;
}

charge = clamp(charge, 0, 1);
```

实际伸手距离：

```js
actualReach = minReachDistance + charge * (maxReachDistance - minReachDistance);
```

目标距离：

```js
targetDistance = distance(currentHold, targetHold);
```

判定：

```js
distanceError = actualReach - targetDistance;
```

- `distanceError < -reachTolerance`：力量不足。
- `Math.abs(distanceError) <= reachTolerance`：成功抓住。
- `distanceError > reachTolerance`：力量过大。

支撑岩点不会进入蓄力判定，只用于手脚支撑。

## 无限岩点生成

路线岩点和支撑岩点都使用世界坐标。每个新路线岩点基于前一个路线岩点向上生成，并验证两点距离处于可达范围内。随机多次失败后会使用确定性的安全回退位置，避免生成理论上无法到达的目标。

每个路线岩点附近会生成若干支撑岩点。支撑岩点只影响姿态和画面，不参与目标锁定、蓄力判定或得分。

离开摄像机下方较远且未被保护的旧岩点会被清理，摄像机上方始终保持足够的岩点缓冲。

## 视觉主题

当前视觉参考明亮的室内攀岩馆：浅冰蓝、白色、淡青蓝和少量粉色分割线。所有元素都由 Canvas 实时绘制，没有把参考图作为静态背景。

墙面由 `drawWall()`、`drawWallPanels()`、`drawWallBoltHoles()` 和 `drawWallTexture()` 绘制。几何墙板按照世界坐标分段重复生成，随摄像机移动，不会固定在屏幕 UI 层。

岩壁左侧绘制白色高度标尺，刻度使用世界坐标和 `pixelsPerMeter = 100` 对齐：每 100px 为 1m，半米显示短刻度，整米显示米数。标尺随 `camera.y` 一起移动，不是固定 HUD。

岩点由 `drawHold()` 统一绘制，当前主攀爬线路优先使用 `assets/pink_climbing_holds_complete_assets/individual_png` 内的粉色透明 PNG；非线路/装饰支撑岩点默认使用 `assets/light_purple_climbing_holds_complete_assets/individual_png` 内的浅紫色透明 PNG。会作为脚点的支撑岩点在生成时就归入粉色线路，小人选脚时只会从粉色线路岩点中选择，可以跳过部分粉色冗余脚点，但不会踩紫色岩点。两类岩点都会按岩点 ID 稳定选择不同图片，保证每一帧不闪烁；如果图片加载失败，会回退到原来的 Canvas 矢量岩点。

岩点图片视觉尺寸和玩法判定尺寸分离。蓄力、抓点、脚点选择仍然使用原有 `hold.radius`、世界坐标和目标距离，PNG 只替换显示效果。

角色绘制保留现有 IK、固定肢体长度和手脚接触逻辑，只替换外观：深色卷发、粉色上衣、青色短裤、蓝色安全带、粉袋和深色攀岩鞋。角色面向墙壁，不绘制眼睛。

成功抓住新岩点并进入休息态后，角色会以当前手点为轻微摆动轴做左右晃动。单手休息过场会表现为手臂先弯曲收回，再把小臂向外甩到伸直。休息期间还会每 2 秒左右轮流用左手或右手去粉袋附近做“抓两下、甩一下”的搓粉动作，动作只影响视觉 IK 目标，不修改真实手脚接触数据。

成功抓点后的身体跟随阶段会同步插值双脚到附近脚点，让手、身体和腿部看起来一起完成攀爬动作。脚点接触仍然在同步移动结束后统一确认，不改变蓄力判定或路线生成。双脚不会完全同速抵达：落点较高的脚会先到，落点较低的脚会根据高度差和移动距离差延后到达。

摄像机跟随也会在成功抓点后的身体和脚部同步移动阶段提前启动。进入 `READY` 前如果镜头已经到位，就直接开放下一次输入；只有极少量残余镜头移动时才进入独立的 `CAMERA_FOLLOW` 收尾。

失败后会立即显示“本轮结束”轻量标签，标签内显示本次攀爬岩点数和最高攀爬岩点纪录，并允许点击屏幕马上开始下一轮；角色动画仍会短暂失衡，然后进入类似攀岩馆自动保护器的匀速下放。下放过程中角色转向面朝屏幕、手脚下垂、低头失落，直到落到屏幕下边界外再进入完整结算。

绳索从 Canvas 顶部边缘直接进入画面并连接到安全带，不绘制顶部锚点、快挂或金属件。绳索使用两层浅蓝线条表现简单编织感。

顶部 HUD 是一条半透明蓝色横栏，只显示当前高度和最高高度纪录；岩点数量和最高岩点纪录移到失败掉落时的本轮结束弹窗中。底部蓄力条使用白色粗圆角外框和绿、黄、橙、红渐变，当前进度由 `game.charge` 控制裁剪宽度。松手时会直接使用当下 `game.charge` 对应的伸手距离进行判定，因此回落阶段松手也会得到较小的出手力量。

## 当前已完成

- 竖屏逻辑分辨率 `390x844`，支持窗口缩放和 `devicePixelRatio`。
- 自动锁定下一个路线岩点，不需要方向控制。
- 按住蓄力、松开出手、成功抓住、力量不足、力量过大、失败结算、点击重开。
- 路线岩点和支撑岩点分离，路线岩点也可作为脚点候选。
- 三点/四点支撑、正身/侧身休息姿态、单手悬吊甩手过场。
- 抓点后的休息态轻微左右晃动，并周期性触发粉袋甩手视觉动作。
- 成功抓点后的身体跟随阶段同步移动双脚，高脚先到、低脚后到，减少后置补脚的割裂感。
- 摄像机上移提前到身体和脚部同步移动阶段，减少进入 `READY` 前的等待。
- 岩壁左侧有白色米数标尺，按世界坐标 1:1 跟随镜头移动。
- 固定长度两段手臂和腿部绘制。
- 失败后立即显示本轮结束标签并允许快速重开，同时自动保护器式匀速下放动画继续播放。
- 历史最高岩点数和最高高度通过 `localStorage` 保存；存储不可用时不影响游戏。
- 轻量换装 MVP：开始界面的“换装商城”入口可切换发型、配饰、上衣、裤子和粉袋，并通过 `localStorage` 保存换装结果。
- 明亮攀岩墙视觉、动态几何墙板、粉色线路岩点和浅紫色非线路岩点、插画风角色、半透明顶部高度栏和底部蓄力条。
- `DEBUG` 默认关闭，打开常量后可显示状态、距离、误差、容错、接触点和摄像机信息。

## 当前限制

- 没有真实物理、真实绳索模拟或完整人体 IK。
- 角色动作仍然是程序插值动画，在极端随机布局下可能出现不够自然的姿态。
- 角色仍是 Canvas 矢量绘制；岩点已接入 PNG 资产，但尺寸、锚点和视觉层级仍是原型级适配。
- 目前只有一次失败即结束的无限挑战模式。
- 墙面几何图案是轻量伪随机分段，不是完整关卡编辑器。

## 迁移到手机小程序时需要替换

- 将 DOM Canvas 获取和浏览器 Pointer Events 替换为小程序 Canvas 与触摸事件。
- 将 `requestAnimationFrame` 调度替换为小程序可用的帧循环。
- 将 `localStorage` 封装替换为小程序本地存储 API。
- 根据小程序 Canvas API 调整上下文初始化、DPR 和尺寸缩放逻辑。

## 抓握音效系统

当前抓握成功音效只使用一个基础文件：`assets/audio/grab-success.ogg`。游戏会在加载阶段通过 Web Audio API 预加载这个文件，解码后缓存为 `AudioBuffer`，成功抓握时用新的 `AudioBufferSourceNode` 播放，因此多个短音效可以快速连续触发，不会互相抢占播放进度。

`lucky`、`good`、`perfect` 三档反馈使用同一个音频文件，通过 `playbackRate` 和音量区分：

- `lucky / 惊险`：`playbackRate = 0.84`，`volume = 0.75`
- `good / 不错`：`playbackRate = 1.0`，`volume = 0.8`
- `perfect / 精准`：`playbackRate = 1.18`，`volume = 0.9`

连击会在当前等级的基础上继续提高音高：`comboPitchBonus = Math.min(comboCount, 6) * 0.04`，最终 `playbackRate` 不超过 `1.35`。这样连续精准抓点时，声音会逐步变亮、变高，形成连击感。

短促抓握音效适合实时变调，因为持续时间短，轻微改变 `playbackRate` 不会明显破坏节奏或音色，反而能用很低的资源成本表达不同反馈等级。背景音乐这类长音频不适合用同样方式频繁变调。

如需替换抓握音效，直接替换 `assets/audio/grab-success.ogg` 即可。建议保持文件较短、起音清楚、无长尾混响，并继续使用浏览器兼容的短音频。替换后刷新页面，游戏会重新加载并解码新的基础音效。
